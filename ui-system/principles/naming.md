# Naming（组件命名原则）

Design System 中组件的命名直接影响可理解性、可扩展性和可维护性。不好的命名会制造虚假的边界，限制未来的演进方向。

> **适用范围**: 所有组件、Design Token、Template 的命名
>
> **关联原则**: [Compose, Don't Duplicate](./compose-dont-duplicate.md) · [语义色统一](./semantic-color-unification.md)

---

## 一、核心规则：命名三问

在命名任何组件之前，先回答三个问题：

| # | 问题 | 反模式示例 | 正确示例 |
|---|------|-----------|---------|
| 1 | **组件名描述的是"是什么"还是"从哪来"？** | ~~Markdown~~（绑定 .md 格式） | Prose（描述排版能力） |
| 2 | **这个名字能容纳未来 3 年的扩展吗？** | ~~Prose~~（只覆盖排版，装不下图表和 AI 内容） | Content（内容呈现领域） |
| 3 | **它和其他组件的关系是同级还是包含？** | ~~Prose.md~~（与其他组件平级，却要覆盖整个内容领域） | Content/Prose（命名空间 + Foundation 模块） |

## 二、命名层级

### 原子组件

描述"是什么"——用最精准的名词描述组件本身的形态或能力。

| 组件名 | 描述的是什么 | 不是描述什么 |
|--------|-------------|-------------|
| Button | 可点击的交互元素 | 不描述点击后发生什么 |
| Checkbox | 多选控件 | 不描述数据从哪来 |
| Input | 文本输入区域 | 不描述输入什么内容 |
| Card | 内容容器 | 不描述装什么内容 |
| Badge | 状态标签 | 不描述标签背后的业务含义 |
| Toast | 浮动通知 | 不描述通知的具体内容 |

**规则**: 组件名 = 形态名词。Button、Card、Input——用户一眼就知道这是什么 UI 元素。

### 命名空间

当一个领域可能包含多个子模块时，使用目录作为命名空间。

```
Content/          ← 命名空间（内容呈现领域）
├── Prose         ← Foundation 模块（基础排版）
├── CodeBlock     ← 语法高亮（未来）
├── Callout       ← 提示块（未来）
└── AI            ← AI 内容扩展（未来）
```

**规则**: 命名空间名 = 领域名词。命名空间不是组件——`compose-dont-duplicate.md` 中引用具体模块（`Content/Prose`），而非命名空间本身。

### 变体

变体名描述使用场景或尺寸级别。

| 组件 | 变体 | 命名模式 |
|------|------|---------|
| Prose | Base / Compact / Inline | 场景级别 |
| Button | Solid / Ghost / Outline | 视觉风格 |
| Card | Product / Stat / Content | 使用场景 |
| Input | Base / With Button / Form | 组合模式 |

**规则**: 变体名 = 场景或风格形容词。

## 三、反模式清单

| 反模式 | 为什么危险 | 本次教训 |
|--------|-----------|---------|
| **以数据源命名** | 当数据源变化时名称失效。~~Markdown~~ 暗示内容总是 .md 文件——但 CMS 富文本、AI 输出、Notion 导入都不是 Markdown | Markdown → Prose |
| **以单一能力命名** | 当领域扩展时名称成为限制。~~Prose~~ 准确描述了排版能力，但无法收纳图表（Mermaid）、公式（KaTeX）、AI 交互（Citation） | Prose → Content/Prose |
| **扁平化组件** | 当子模块增多时，根目录充斥 50+ 个 .md 文件，难以导航 | Content/ 命名空间 |
| **以业务名词命名** | 业务名词会变，组件能力不变。~~CheckoutForm~~ → 如果支付流程变了呢？ | 应命名为 PaymentForm 或直接复用 Form |
| **缩写命名** | 降低可理解性，新人需要额外学习。~~Btn~~、~~Nav~~、~~Pagi~~ | 永远用全称 |

## 四、命名检查流程

在创建新组件时，按以下顺序检查：

```
1. 这是什么 UI 元素？
   → 用形态名词命名（Button、Card、Input...）

2. 它属于哪个领域？
   → 如果领域已有或可能有多于一个子模块，创建命名空间

3. 它绑定特定格式/来源吗？
   → 如果是，抽象一层。描述"是什么"而非"从哪来"

4. 5 年后这个名字还适用吗？
   → 如果犹豫，说明名字不够抽象
```

## 五、与 compose-dont-duplicate 的关系

| 关注点 | Naming | Compose, Don't Duplicate |
|--------|--------|--------------------------|
| 解决的问题 | 组件叫什么 | 组件如何构建 |
| 错误方式 | 以数据源命名 | 每个组件自己写一套样式 |
| 正确方式 | 以形态/领域命名 | 新组件 = 已有组件 + 新布局 |
| 本次案例 | Markdown → Prose → Content | Prose 引用 spacing.md + colors.md，不引用其他组件 |

两者互补：好的命名让依赖关系更清晰，清晰的依赖关系反过来验证命名是否正确。
